## 2026-07-23 14:41:35 -07:00 — Working-window handoff

Reviewed against current checkout commit `7707968`.

Corrections the oracle should carry into later handoffs:

- The current rewrite has no headless asset-generation contract. The private
  `IsolateSession._generation` value is GUI display lifecycle state and must not
  be placed in reusable source requests, batches, or scientific identity.
- `ActiveAsset` is immutable but shares a module with the PyQt
  `ActiveAssetController`. A genuinely headless contract must not import that
  module; use a headless request/reference plus a thin GUI snapshot adapter.
- The active asset snapshot does not currently contain the active media's
  `content_sha256`, although the asset sidecar does. Reading the recorded
  sidecar hash is not the same as verifying the current media bytes. Ordinary
  resolution may compare cheap current facts such as size, dimensions, and
  rational fps, but explicit content verification remains separate.
- Scientific source requests should require an existing registered sidecar.
  They must not silently initialize a raw video merely to manufacture stable
  identity.
- The current media API exposes only FFmpeg-produced `rgb24` bytes through
  `MediaSession.read_frame_rgb`. It has no general plane ids, plane registry,
  luma/alpha/native-plane delivery, or multi-plane batches.
- Isolate display explicitly requests a width-limited preview. Scientific
  source handoffs must require native decode and must not reuse that preview as
  evidence.
- `MediaSession.frame_count` may come from decoded, packet, container, or
  duration-derived counts. Later handoffs must retain this provenance and must
  not call every admitted extent verified.
- The current `MediaSession` owns one mutable FFmpeg cursor. A scientific
  iterator must use a request-local session rather than sharing the live player
  session.
- For a source with no consumer, the smallest bounded contract is synchronous
  iteration with natural backpressure. Do not require a worker, producer queue,
  GUI latest-only mailbox, or cross-thread cancellation until a real channel
  consumes the source.
- A source stream must make its final outcome accessible after ordinary
  iteration and must close request-local media on normal completion,
  cancellation, failure, explicit close, and early consumer exit.
- Current short RGB reads all become `FRAME_DECODE_FAILED`; the media layer does
  not preserve a structured clean-EOF versus decode-error reason. Do not require
  the working-window layer to guess truncation from message text or stderr.
- Decoded/delivered source span is not examined scientific coverage. Reserve
  `examined`, `processed`, `quiet`, and `detected` for completed scientific
  nodes.
- The current backend returns a complete exact-sized frame or an error. General
  spatial validity masks and coverage artifact types are not yet justified.
- A standalone headless diagnostic cannot observe another running Qt process's
  private current window or generation without an explicitly designed bridge.
  Until a real GUI consumer exists, use explicit diagnostic inputs and a pure
  GUI request-snapshot test.

## 2026-07-23 14:53:33 -07:00 — Working-grid handoff

Reviewed against current checkout commit `4b09232`.

Corrections the rewrite should carry into its assessment of the working-grid
handoff:

- Commit `4b09232` contains the corrected milestone-3 documentation, but no
  working-window implementation. Milestone 4 must remain queued until milestone
  3 is implemented and accepted, then rerun its divergence check against the new
  headless package and GUI snapshot seams.
- `ActiveAsset.width` and `height` are registered native dimensions, but
  `ActiveAsset` still lives in a PyQt-bearing module. Pass primitive native
  dimensions into the headless grid resolver; do not import `ActiveAsset` there.
- `IsolatePlayer.frame_size` is the capped display-preview size, not the native
  source extent. It must not be used to resolve working dimensions or block
  geometry.
- `IsolatePlayer` currently exposes only `image_rect()` for its letterboxed
  preview. It has no native-source-coordinate transform. Map grid boundaries
  directly by normalized working coordinates into `image_rect()` rather than
  creating a general display adapter.
- The normalized current-player mapping is:
  `view_x = left + (working_x/work_width)*drawn_width` and equivalently for
  `y`. The outside grid boundary therefore matches `image_rect()` even when the
  preview dimensions were capped or rounded.
- `IsolatePlayer` has no overlay input, zoom, or overlay-hide shortcut. Add one
  grid-specific optional paint input; do not copy the interactive Replicates
  `FrameView` or introduce a general overlay framework. Do not add zoom or a
  shortcut for this milestone.
- `IsolateSession` currently owns temporal playback, a mutable display
  `MediaSession`, timers, and the display decoder thread. Keep the grid resolver
  Qt-independent and do not make it part of this media-owning session merely for
  convenience. The smallest current GUI owner is `IsolateTab`; milestone 3's
  later request-snapshot method remained temporal and did not replace that
  owner.
- The rewrite currently has no downsample/block settings or persistence policy.
  The original review proposed initializing `1.0` and auto once per Isolate-tab
  session and retaining those intents across asset changes. The post-milestone-3
  refresh below supersedes that proposal as an assumed policy: retain versus
  reset requires a user decision. Do not add sidecars or `QSettings` either way.
- The loaded player already has `IsolateDecodeThread`. “No worker or decode on
  grid change” means no additional grid worker and no additional request to the
  existing display decoder, not that the application has zero threads.
- Current temporal controls use `can_loop` and are disabled on one-frame assets.
  Grid controls are spatial and must instead be enabled from valid active-asset
  geometry so one-frame assets remain supported.
- Dense-grid suppression belongs inside player presentation. It must not change
  the resolved grid, silently increase block size, or feed widget resize back
  into scientific settings.
- The right-hand channel placeholder is still intentionally empty. Grid
  controls should not replace it or introduce a fake channel; use a compact
  separate control/readout area in the existing Isolate layout.

## 2026-07-23 15:14:42 -07:00 — Working-grid post-milestone-3 refresh

Reviewed and updated against current checkout commit `01b256f`.

Corrections the oracle should carry into later handoffs:

- The milestone-4 review's central checkout description is now stale.
  Milestone 3 was implemented in `5bffca7`; the rewrite now has a Qt-free
  `WorkingWindowRequest`, `ResolvedWorkingWindow`, native `rgb24` batches,
  request-local synchronous streaming, explicit extent provenance and final
  outcomes, and a GUI request-snapshot adapter.
- `WorkingWindowRequest` contains registered asset identity and half-open
  temporal bounds. It intentionally contains neither source dimensions nor
  spatial tuning. `ResolvedWorkingWindow` contains probed native width and
  height, but obtaining it opens a request-local media session. Milestone 4
  must not open a working-window source merely to resolve or repaint geometry.
- The implemented GUI adapter is
  `IsolateSession.snapshot_working_window_request()`. It snapshots the current
  asset identity and temporal window without opening scientific media, but it
  does not establish a general Isolate settings controller. Grid intent should
  remain a plain Qt-independent value owned by `IsolateTab`, not be added to
  the playback/media lifecycle of `IsolateSession`.
- `ActiveAsset` now carries registered native `width`, `height`,
  `content_sha256`, and sidecar/media references. It still lives in a
  PyQt-bearing module. Pass primitive width and height from the active snapshot
  into the headless grid resolver; do not import `ActiveAsset` into the
  resolver.
- For immediate UI geometry, registered `ActiveAsset.width,height` are the
  available no-I/O extent. They are recorded facts, not freshly
  content-verified evidence. When a later channel combines a resolved grid with
  pixels, it should require equality with
  `ResolvedWorkingWindow.width,height` and retain that source's recorded
  identity and extent provenance.
- The rewrite still has no working-grid type, downsample setting, block
  setting, spatial-settings persistence, or accepted spatial dimension
  rounding rule. The handoff's Python `round` rule remains available, but must
  be defined once in the grid resolver and pinned at half ties.
- The review's proposed session-local retention across active-asset switches is
  not current rewrite behavior and has not been accepted as a product decision.
  Whether asset switches retain `downsample` and block intent or reset them is
  the only true behavior choice still requiring the user; either policy can
  re-resolve safely against the new primitive native dimensions.
- The current `IsolatePlayer` remains a capped-preview painter with one
  letterboxed `image_rect()` and no zoom or overlay seam. The review's
  normalized mapping directly into `image_rect()` remains the smallest correct
  projection. No milestone-3 implementation created a display adapter that the
  grid should reuse.
- Grid control changes must create no `WorkingWindowStream`, no
  `MediaSession`, no grid worker, and no request to the existing
  `IsolateDecodeThread`. The existing display thread does not violate this
  rule; only additional work caused by a grid change would.
- The review's out-of-scope implementation findings remain separate from
  milestone 4. In particular, milestone 3 did not repair the stale GUI decode
  error path, GUI decoder start/interrupt shutdown race, derivation's
  metadata-only trust of recorded content identity, synchronous first-open
  registration, capped-preview aspect distortion, or incomplete benchmark
  provenance. Grid code must not absorb those responsibilities.
- Focused post-refresh validation passed:
  `tests/test_working_window.py`, `tests/test_isolate_player.py`, and
  `tests/test_domain_and_services.py` report `49 passed in 10.46s` with Qt
  offscreen. This supersedes the review's pre-milestone-3 `34 passed` evidence
  for current-checkout assessment, while leaving the handoff's manual
  acceptance path outstanding.
