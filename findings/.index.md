# Findings index

## Results of hypothesis testing

- [Media viewer baseline and first performance pass](2026-07-23-media-viewer-performance.md)
  — tests zero-copy Qt image ownership, explicit 1280-pixel Isolate preview
  decoding, and settled pointer scrubbing against the existing full-resolution
  media path.

## Counterintuitive findings

- The 5.3K source's compressed-frame decode was not the dominant steady-state
  cost by itself. Producing and transferring full-resolution RGB made the
  viewer path about ten times slower than the explicit 1280-pixel display
  representation. See
  [Media viewer baseline and first performance pass](2026-07-23-media-viewer-performance.md).

## Patterns to follow

- Keep full-resolution decoding as the media-service default and make reduced
  display representations explicit at the GUI call site. See
  [Media viewer baseline and first performance pass](2026-07-23-media-viewer-performance.md).
- Promote reusable user-facing estimates into package-owned diagnostics with
  supported CLI or UI reporting. See
  [Reusable benchmarks are product diagnostics](../docs/decisions/001-reusable-benchmarks-are-product-diagnostics.md).

## Antipatterns

- Do not copy a frame into a Qt-owned allocation when the widget can retain the
  immutable Python buffer for the complete `QImage` lifetime. See
  [Media viewer baseline and first performance pass](2026-07-23-media-viewer-performance.md).
