# Media viewer baseline and first performance pass

## BLUF

On the representative 5312×2988, 59.94 fps project source, the retained changes
reduced native-window steady-state Isolate decode-to-paint latency from
`85.36 ms` to `8.47 ms` median, a `90.1%` reduction. The result is below the
asset's `16.68 ms` native-frame budget in this short benchmark.

Three independently measured changes contributed:

1. Retaining immutable RGB bytes for the `QImage` lifetime instead of copying
   every delivered frame reduced native steady-state Qt image installation from
   `7.86 ms` to `1.20 ms` median (`84.8%`).
2. Explicitly requesting a 1280×720 Isolate display representation reduced
   service-level steady sequential reads from `74.47 ms` to `5.55 ms` median
   (`92.5%`) and random seeks from `441.25 ms` to `267.11 ms` median (`39.5%`).
3. Settling pointer-drag requests before decode reduced the one measured final
   scrub-release latency from `367.43 ms` to `165.92 ms` (`54.8%`), while both
   runs painted only the final requested scrub frame.

The default full-resolution `MediaSession` path remains independently
requestable and showed no material performance change. Exact same-backend frame
hashes on the numbered lossless seek fixture still matched after the changes.
All `78` tests passed.

- Platform: Windows `10.0.26200`, x86-64.

- Python: `3.11.9` from the repository `.venv`.

- Operating-system filesystem caches were not flushed. New-session measurements

### Measurement boundaries

The service benchmark separately records:

- FFprobe/session construction.
- Open through first decoded frame.
- Decoder startup.
- Adjacent sequential raw-RGB reads.
- Non-adjacent random-seek reads.

The viewer benchmark synchronously records:

- Sequential decode.
- `QImage` installation.
- Paint.
- Decode through completed synchronous repaint.
- Repainting an unchanged frame.

The first decoded frame includes FFmpeg startup and is reported separately from
steady adjacent frames in the interpretation. The native viewer was shown,
not minimized, and used Qt's `windows` platform plugin. The tool could not prove
that no other desktop window obscured it during every paint.

The interaction benchmark sends 100 positions over approximately one second,
then measures release until the final frame is delivered. It also switches from
the large source to the lossless fixture and checks final-frame dimensions and
clean session state after close.

## Comparison

The baseline showed that an adjacent full-resolution source frame spent about
`74.98 ms` in decode/output and another `7.86 ms` installing the Qt image. The
first candidate removed the redundant Qt-owned copy but could not make
59.94 fps playback feasible because full-resolution RGB production remained the
dominant stage.

The second candidate kept native media decoding available while asking only the
Isolate display path for 1280×720 RGB. This reduced the amount of raw RGB
transferred per frame from approximately `47.6 MB` to `2.76 MB`, about
`17.2×` less pixel data. The measured complete native-window steady path then
fell below the source's frame budget.

The scrub change addressed a separate random-access problem. The latest-only
worker already prevented obsolete frames from painting, but it could begin an
obsolete seek during a pointer drag. Deferring pointer-drag decode until the
60 ms settle timer or mouse release removed that avoidable seek without
debouncing playback, programmatic timeline seeks, or keyboard stepping.

## Interpretation

Retain all three changes:

1. Retain the immutable RGB bytes beside each `QImage`; this gives the Qt view a
   valid buffer lifetime without an extra full-frame allocation and copy.
2. Retain the explicit 1280-pixel Isolate display request. It is the change that
   addresses the dominant measured steady-state cost. Keep native resolution as
   the `MediaSession` default and do not silently feed preview pixels into CLI,
   Replicates focus behavior, or future scientific computation.
3. Retain pointer-only scrub settlement. It improves final-request latency while
   preserving latest-request correctness and immediate non-drag controls.

Do not add prefetch, a large cache, hardware decoding, or shared cross-tab
decoder ownership from this finding. The current evidence does not justify
their lifecycle and memory complexity.

Open-to-first-frame and random seeking remain much slower than steady playback.
Those are now clearly subprocess/startup and random-access questions rather
than ordinary sequential throughput questions. They are suitable follow-up
hypotheses, not reasons to stack another speculative change into this pass.

## Potential ways this finding could be invalid now or later

- Measurements came from one Windows machine, one FFmpeg build, and one main
  project source. Other codecs, hardware, storage, or operating systems may
  change both absolute latency and the best display width.
- Viewer samples contain only eight frames. Median separation is large, but
  longer native-window runs are needed for stable tail estimates, thermal
  behavior, and missed-deadline counts.
- The operating-system cache was not flushed. Open timings describe new
  sessions over potentially cached media, not cold storage.
- CPU utilization and peak resident memory were not collected in this first
  pass. Reduced raw-frame size should reduce traffic, but no memory reduction is
  claimed without measurement.
- Native painting used a real window and the Windows Qt plugin, but the harness
  could not guarantee that the window was never obscured by another desktop
  surface.
- The synchronous viewer benchmark omits normal queued-signal scheduling. The
  separate interaction benchmark exercises the actual `IsolateSession` worker,
  but it was run offscreen and only once per before/after state.
- Exact hash verification used a small lossless fixture and the same FFmpeg
  backend. It does not establish cross-backend lossy pixel equivalence or prove
  temporal identity at every frame of the project source.
- A fixed 1280-pixel maximum is appropriate for the measured 960×540,
  device-pixel-ratio-1 display. A future high-DPI, zoom, or native-detail
  Isolate view may need an explicit larger representation and cache key.
- The project source is approximately 59.94 fps. Results should not be assumed
  to establish native-rate behavior for 120 fps media.
- Future overlays or channel visuals may make paint the dominant stage even
  though it is not dominant now. They require their own stage measurements.
